# how-to-manage-memory — 장기메모리와 진행도

휘발성 세션들이 모여 하나의 큰 작업을 끝낸다. 각 세션은 시작할 때 자기 머릿속이 비어 있으므로,
**싼 비용으로** 두 가지를 알 수 있어야 한다 — "전체가 지금 어디까지 왔나(where are we)"와
"내가 맡은 건 어느 위치의 무엇인가(where am I)". 이 둘을 파일에 남겨 세션 간에 잇는 것이
장기메모리의 책임이다.

핵심은 **읽기 비용**이다. Volume이 클수록 전체 메모리를 매 세션 다 읽는 건 불가능하다. 그래서
메모리는 "한 세션이 자기 위치를 파악하는 데 읽어야 하는 양이 작업 전체 크기와 거의 무관하게
일정하도록(read-cost-first)" 구조화한다. 이 원리가 아래 모든 포맷·보관 결정을 끌고 간다.

## 이 문서가 다루는 것 — 메모리 store만

이 문서는 **장기메모리라는 저장소 자체**(무엇을, 어떤 포맷으로, 언제부터 언제까지 남기고,
세션이 그것을 어떻게 읽고 갱신하는지)만 다룬다. 무엇을 명확화할지, 어떻게 쪼갤지, 개별 세션에
어떤 메시지를 넘길지는 각자의 책임이며 여기서 가정하지 않는다. 메모리는 그 산출물들 —
확정된 요구사항(spec), 쪼개진 노드, 세션이 남긴 결과 — 을 **generic하게 받아** 보관하고
조회 가능하게 만든다. 그 산출물이 어떻게 만들어졌는지는 메모리의 관심사가 아니다.

## 원칙

1. **읽기 비용으로 설계한다.** Volume이 커도 한 세션이 자기 위치를 파악하는 비용은 거의 일정해야
   한다. 메모리는 "조상 경로 + 내 노드 + 형제 상태 요약"처럼 **국소적으로(tree-local)** 읽히도록
   구조화한다. 전체 트리·전체 history를 매번 강제로 읽게 만들면 작업이 커질수록 비용이 폭발한다.

2. **변경 빈도로 계층을 가른다.** 거의 안 변하는 것(목표·범위·제약), 매 세션 변하는 것(진행
   상태), 한 번 쓰고 닫히는 것(완료 노드의 결과)을 한 파일에 섞지 않는다. 섞으면 자주 변하는 쪽이
   안정된 쪽을 오염시키고 diff가 불필요하게 커진다.

3. **상태는 덮어쓰고, 사건은 쌓는다.** 현재 진행 상태(어디까지 됐나)는 항상 현재만 —
   덮어쓴다(mutable). 일어난 결정·실패·발견(history)은 지우지 않고 append한다. 둘을 한 파일에
   섞으면 상태 파일이 history로 부풀거나 history가 덮여 사라진다.

4. **상태의 진실은 한 곳에만(single source of truth).** 같은 사실(노드 상태·진행도)을 두 곳에
   쓰면 반드시 갈라진다. 진행 상태는 단 한 곳에서만 갱신하고, 나머지는 그것을 참조한다.

5. **"왜"와 "막다른 길"을 남긴다.** 산출물(코드·파일)은 "무엇을 했는가"를 스스로 증언한다.
   하지만 "왜 그렇게 결정했는가"와 "무엇을 시도했다 왜 실패했는가"는 어디에도 안 남는다. 이것이
   메모리의 가장 비싼 자산이다 — 실패 기록이 없으면 다음 세션이 같은 막다른 길을 다시 판다.

6. **쌓지 말고 가지치기한다(curate).** 무한정 쌓으면 신호가 잡음에 묻힌다. 노드가 닫히면 그 안의
   상세 로그는 결과 요약 한 덩어리로 압축(roll-up)해 hot 경로에서 빼고, 원본은 버리지 말고
   아카이브로 옮긴다. 살아있는 frontier만 상세하게 유지한다.

7. **메모리는 작업 트리를 미러링한다.** Volume이 클수록 평면 단일 파일은 깨진다. 분할이 n-depth
   트리면 메모리도 그 트리를 따라가고, 각 기록은 자기가 어느 노드인지 식별자를 갖는다 — 그래야
   세션이 "내 위치"를 국소적으로 조회할 수 있다.

## 메모리 구조 — 5계층

각 계층은 **무엇을 / 어떤 포맷으로 / 언제부터 언제까지** 남기는지가 다르다.

- **① Charter (목표층, 거의 불변).** 전체 작업의 목표·범위·제약·수용기준. 요구사항 명확화의
  산출물을 받아 시작 시 1회 쓰고 거의 손대지 않는다(범위가 실제로 바뀔 때만, 그건 큰 사건이다).
  모든 세션이 가장 먼저 읽는 north star. 산문, 작게 유지.

- **② Map (지도층, 느리게 변함).** 쪼개진 노드 트리. 각 노드는 식별자·목표·경계·검증법·의존,
  그리고 **상태**를 갖는다. 상태가 "where are we"의 single source of truth다. 한 세션은 여기서
  frontier와 자기 노드·조상 경로·형제/의존 노드의 상태를 찾는다. 노드의 "어떻게"(구현 상세)는
  여기 두지 않는다 — 부피가 터진다. 첫 분할 때 생성되고 매 세션 상태가 갱신되며 전 기간 hot.

- **③ 노드 작업기록 (국소, append-only).** 노드마다 작은 기록: 내린 결정과 이유, 시도했다 실패한
  접근과 왜, 그 노드의 open question. 국소성이 핵심 — 한 세션은 자기 노드와 조상의 기록만 읽지,
  전체 노드 기록을 읽지 않는다. 노드 착수 시 생성, 열려 있는 동안만 hot.

- **④ 노드 종결 요약 (immutable).** 노드가 완료될 때 ③을 압축해 만드는 한 덩어리: 무엇을
  산출했나(참조로, 중복 금지), 남은 caveat, 부모·의존 노드가 알아야 할 것. 완료 시 1회 쓰고 이후
  불변. 부모·형제·의존 노드가 읽고 곧 cold.

- **⑤ 전역 교훈/규약 (curated).** 모두에게 영향을 주는 횡단 발견 — 코드베이스 규약, 프로젝트
  전역의 막다른 길, 반복되는 제약. opinionated하게, 프로젝트 고유의 것만, 작게 유지한다(일반
  프로그래밍 상식은 넣지 않는다). 발견이 한 노드를 넘어 일반화될 때 append, 주기적으로 가지치기.

### 위치 — 워크스페이스 루트의 `.longtask/`

메모리 루트는 작업이 진행되는 **워크스페이스 루트 기준 `.longtask/`** 에 둔다. 작업 대상
코드베이스와 같은 곳에 살아야 그 작업의 메모리이기 때문이다. 메모리는 **프로젝트마다 독립**이며,
한 워크스페이스의 진행도가 다른 워크스페이스로 새지 않는다.

플러그인 자신의 코드(method 문서 등)가 사는 설치 경로와는 **명확히 구분**한다. 그곳은 모든
워크스페이스가 공유하는 읽기 전용 코드이고 업데이트 시 덮어써지므로, 메모리를 그곳에 쓰지 않는다.

```
.longtask/
  charter.md            ① 목표·범위·제약·수용기준
  map.md                ② 노드 트리 + 상태 (전역 single source of truth)
  lessons.md            ⑤ 전역 교훈/규약
  nodes/
    <id>/record.md      ③ 노드별 작업기록 (append-only)
    <id>/result.md      ④ 노드별 종결 요약 (닫힐 때)
  archive/
    <id>/record.md      닫힌 노드의 verbose 원본 (영구 보존, hot 경로 밖)
```

**포맷.** 기록(③④⑤)은 사람가독 산문 Markdown. ②Map은 Markdown이되 상태·의존을 스캔 가능한
구조 필드로 — 사람도 기계도 읽을 수 있게:

```
## n1.2 인증 모듈   status: in_progress
- goal: 로그인/세션 발급
- blocked_by: [n1.1]
- verify: 로그인 e2e 통과
```

**노드 상태 어휘.** `미확정`(추가 탐색 필요) · `분할됨`(자식 노드 있음) · `구현대기` ·
`진행중` · `완료` · `막힘`(blocked). frontier = 아직 `완료`/`분할됨`이 아닌, 지금 손댈 수 있는 노드.

## 절차 — 세션 시작: 위치 파악 (orient)

읽기량이 작업 크기와 무관하게 일정한 것이 목표다. 국소적으로만 읽는다:

1. **① charter** 읽기 — 전체가 무엇인가.
2. **② map**에서 내 노드를 찾는다 — 조상 경로, 내 노드의 목표·경계·검증, 형제·의존 노드의 상태.
3. **③ 내 노드 기록 + 조상 기록 + ⑤ 교훈** 읽기 — 이미 내려진 결정, 실패한 접근, 전역 규약.
4. **여기서 멈춘다.** 전체 트리의 모든 노드, 전체 history, 아카이브는 읽지 않는다(필요할 때만
   해당 노드 것을 소환). 이 bounded read가 설계의 핵심이다.

## 절차 — 세션 종료: 반영 (update)

진실은 한 곳, 상태는 덮어쓰고 사건은 쌓는다:

1. **② map에서 내 노드의 상태만 갱신**한다(single source of truth). 진행 중 발견이 분할을 깨면
   노드를 split/merge하고 그 변경도 여기서.
2. **③ 내 노드 기록에 이번 세션의 결정·실패·발견을 append**한다. 특히 시도했다 실패한 접근과
   그 이유를 반드시 남긴다(원칙 5).
3. **한 노드를 넘어 모두에게 영향을 주는 것은 ⑤ 교훈으로 승격**한다.
4. **노드가 완료면 ③을 ④ 결과 요약으로 roll-up**하고, ③ 원본은 `archive/`로 옮겨 hot 경로에서
   뺀다(삭제하지 않는다 — 막다른 길 히스토리는 영구 보존).
5. 현재 상태 파일(①②)은 작게 유지하고, 서사는 append-only 기록(③⑤)으로 민다.
